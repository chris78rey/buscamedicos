```python
def _can_access_file(user: User, file_record: File) -> bool:
    if user.id == file_record.owner_user_id:
        return True
    try:
        role_codes = [r.code for r in user.roles]
    except Exception:
        role_codes = []
    if any(r in role_codes for r in ["super_admin", "admin_validation"]):
        return True
    return False
```

```python
async def _can_access_file(user: User, file_record: File) -> bool:
    if str(user.id) == str(file_record.owner_user_id):
        return True

    try:
        await user.awaitable_attrs.roles
        role_codes = [r.code for r in user.roles]

        role_codes = []

    if any(r in role_codes for r in ["super_admin", "admin_validation"]):
        return True

    return False
```

#### Fase 2

**Objetivo**
Agregar una verificación mínima para no dejar el fallo oculto.

**Riesgo controlado**
No cambia negocio, solo evita silencios peligrosos.

**Archivo a tocar**
`backend_api/app/routers/files.py`

**Opción conservadora recomendada**

En vez de capturar `Exception` completa, conviene al menos registrar el fallo. Si no se quiere introducir logging todavía, se puede dejar el `except`, pero idealmente no debería tragarse cualquier error silenciosamente.

La versión mínima segura sería:

```python
async def _can_access_file(user: User, file_record: File) -> bool:
    if str(user.id) == str(file_record.owner_user_id):
        return True

    await user.awaitable_attrs.roles
    role_codes = [r.code for r in user.roles]

    if any(r in role_codes for r in ["super_admin", "admin_validation"]):
        return True

    return False
```

Esta versión es mejor porque, si la carga de roles falla, el error se ve y no se maquilla como “acceso denegado”.
