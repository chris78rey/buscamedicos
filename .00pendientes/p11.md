ACTUAR COMO DESARROLLADOR BACKEND SENIOR Y MODIFICAR SOLO EL FLUJO DE VALIDACIÓN ADMINISTRATIVA MANUAL EN:

- backend_api/app/routers/admin.py

OBJETIVO
Permitir que un administrador de validación revise los documentos del médico, descargue el PDF del título, apruebe o rechace documentos y luego apruebe o rechace la solicitud completa.

CONTEXTO ACTUAL
En admin.py ya existen bases de estos endpoints:
- GET /verification-requests
- GET /verification-requests/{request_id}
- POST /verification-requests/{request_id}/assign
- POST /verification-requests/{request_id}/request-correction
- POST /verification-requests/{request_id}/approve
- POST /verification-requests/{request_id}/reject

Pero falta endurecer el flujo y agregar revisión documento por documento.

ALCANCE EXACTO

A. Mantener y corregir:
1. GET /api/v1/admin/verification-requests
- Debe listar solicitudes con:
  - id
  - professional_id
  - status
  - submitted_at
  - assigned_admin_id
  - reviewed_by
  - decision_reason
  - datos básicos del profesional
  - conteo de documentos
  - conteo de aprobados, pendientes, rechazados

2. GET /api/v1/admin/verification-requests/{request_id}
- Debe devolver:
  - solicitud
  - profesional
  - persona
  - documentos
  - download_url por documento

3. POST /api/v1/admin/verification-requests/{request_id}/assign
- Debe poner:
  - assigned_admin_id = usuario actual
  - status = UNDER_REVIEW
- Debe crear VerificationEvent ASSIGNED
- Debe crear AuditEvent

B. Agregar endpoints nuevos
4. POST /api/v1/admin/verification-requests/{request_id}/documents/{document_id}/approve
- Debe aprobar solo ese documento.
- Debe aceptar nota opcional.
- Debe poner:
  - review_status = APPROVED
  - review_notes = nota
- Debe crear VerificationEvent DOCUMENT_APPROVED
- Debe dejar auditoría.

5. POST /api/v1/admin/verification-requests/{request_id}/documents/{document_id}/reject
- Debe rechazar solo ese documento.
- Debe exigir nota.
- Debe poner:
  - review_status = REJECTED
  - review_notes = nota
- Debe crear VerificationEvent DOCUMENT_REJECTED
- Debe dejar auditoría.

C. Endpoints finales de decisión
6. POST /api/v1/admin/verification-requests/{request_id}/request-correction
- Debe exigir reason.
- Debe cambiar:
  - verification.status = NEEDS_CORRECTION
  - professional.onboarding_status = REJECTED o equivalente consistente
  - professional.status = REJECTED
  - professional.is_public_profile_enabled = False
- Debe dejar evento y auditoría.

7. POST /api/v1/admin/verification-requests/{request_id}/approve
- SOLO debe aprobar si existe al menos un documento degree con review_status APPROVED.
- Si no, debe devolver 400.
- Debe cambiar:
  - verification.status = APPROVED
  - decision_at = ahora
  - reviewed_by = admin actual
  - professional.onboarding_status = APPROVED
  - professional.status = ACTIVE
  - professional.is_public_profile_enabled = True
- Debe crear VerificationEvent APPROVED
- Debe crear AuditEvent

8. POST /api/v1/admin/verification-requests/{request_id}/reject
- Debe exigir reason.
- Debe cambiar:
  - verification.status = REJECTED
  - decision_reason = reason
  - professional.onboarding_status = REJECTED
  - professional.status = REJECTED
  - professional.is_public_profile_enabled = False
- Debe crear VerificationEvent REJECTED
- Debe crear AuditEvent

RESTRICCIONES
- Solo super_admin y admin_validation.
- Admin_validation no debe tocar nada clínico.
- No se debe aprobar la solicitud final si el título PDF no fue aprobado antes.
- No se debe hacer validación automática con SENESCYT todavía; la validación sigue siendo manual con descarga del PDF y revisión externa.

PRUEBAS
1. crear solicitud desde cuenta profesional
2. entrar con admin_validation
3. listar solicitudes
4. abrir detalle
5. descargar PDF
6. aprobar documento degree
7. aprobar solicitud
8. revisar que el profesional quede active y público

CRITERIO DE ACEPTACIÓN
- existe trazabilidad completa,
- el admin puede revisar documento por documento,
- el profesional solo se activa al aprobar la solicitud final.
