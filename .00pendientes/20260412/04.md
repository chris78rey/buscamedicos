#### Impacto y riesgos

El problema de la pantalla **Perfil y documentos** del profesional no parece estar en un solo punto. En el estado actual del repo hay **tres desalineaciones reales** entre frontend y backend:

1. El frontend guarda `public_bio`, pero el backend espera `bio_public` en `ProfessionalUpdatePayload`. Eso rompe el guardado del perfil o deja campos sin persistir.
2. El frontend usa tipos de documento que no existen en el enum del backend, por ejemplo `specialty_cert`, `national_id`, `malpractice_insurance` y `other`. El backend solo acepta valores como `degree`, `national_id_front`, `national_id_back`, `registration_certificate`, `selfie_verification`, `cv`, `signed_agreement` y `supporting_document`. Eso rompe el upload y por eso el CRUD de documentos “parece no hacer nada”.
3. La tarjeta de estado usa textos y condiciones que no coinciden con los estados reales del backend. El backend devuelve `submitted`, `under_review`, `approved`, `rejected`, `needs_correction` y `suspended`, pero el frontend evalúa `pending`, `correction_requested` y `draft`. Por eso el estado puede verse mal o quedarse visualmente inconsistente.

También hay un antecedente importante: en una versión anterior del frontend, `useApi.ts` enviaba siempre `Content-Type: application/json`, lo cual dañaba `FormData`. En el snapshot actual eso **ya está corregido** con detección de `FormData`, así que no conviene tocar esa parte otra vez.

#### Fase 3

**Objetivo**

**Riesgo controlado**

`web_frontend/pages/professional/profile.vue`

**Problema puntual**
El backend devuelve `request_status` con estados como `submitted`, `under_review`, `needs_correction`, pero el frontend usa `pending` y `correction_requested`.

**Reemplazar `getStatusColor` por:**

```ts
/* INICIO CAMBIO NUEVO */
function getStatusColor(status: string | null | undefined) {
  switch (status) {
    case 'approved':
      return 'success'
    case 'rejected':
      return 'error'
    case 'submitted':
    case 'under_review':
    case 'pending':
      return 'warning'
    case 'needs_correction':
      return 'info'
    case 'draft':
      return 'grey'
    case 'suspended':
      return 'error'
    default:
      return 'grey'
  }
}
/* FIN CAMBIO NUEVO */
```

**Reemplazar `getStatusText` por:**

```ts
/* INICIO CAMBIO NUEVO */
function getStatusText(status: string | null | undefined) {
  switch (status) {
    case 'approved':
      return 'Aprobado'
    case 'rejected':
      return 'Rechazado'
    case 'submitted':
      return 'Enviado'
    case 'under_review':
      return 'En revisión'
    case 'pending':
      return 'Pendiente'
    case 'needs_correction':
      return 'Corrección requerida'
    case 'draft':
      return 'Borrador'
    case 'suspended':
      return 'Suspendido'
    case null:
    case undefined:
    case '':
      return 'Sin enviar'
    default:
      return status
  }
}
/* FIN CAMBIO NUEVO */
```

**Y reemplazar esta condición:**

```vue
v-if="['draft', 'correction_requested', null].includes(verificationStatus.request_status)"
```

**por esta:**

```vue
v-if="[null, 'needs_correction', 'rejected'].includes(verificationStatus.request_status)"
```

Eso deja el botón visible cuando todavía no hay solicitud o cuando fue rechazada o devuelta para corrección.
