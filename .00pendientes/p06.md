Lo que se observa en el repositorio es que **el CRUD sí está implementado en código** para la agenda del profesional; no parece ser un problema de “faltan endpoints”.

En backend existen rutas para disponibilidades:

* `GET /me/availabilities`
* `POST /me/availabilities`
* `PUT /me/availabilities/{availability_id}`
* `DELETE /me/availabilities/{availability_id}`  

También existen rutas para bloqueos puntuales:

* `GET /me/time-blocks`
* `POST /me/time-blocks`
* `PUT /me/time-blocks/{block_id}`
* `DELETE /me/time-blocks/{block_id}`  

Además, esas rutas sí están registradas en `main.py` bajo `/api/v1/professionals`, por lo que el prefijo también coincide con lo que consume el frontend. 

En frontend, la pantalla `web_frontend/pages/professional/agenda.vue` sí llama a ese CRUD:

* guarda disponibilidades con `createAvailability` y `updateAvailability`
* elimina con `deleteAvailability`
* guarda bloqueos con `createTimeBlock` y `updateTimeBlock`
* elimina con `deleteTimeBlock`  

Y el composable `useProfessionalAgenda.ts` apunta exactamente a esos endpoints.  

Entonces, el diagnóstico más probable es este:

1. **No está fallando el CRUD por ausencia de código, sino por error de ejecución o validación.**
   El backend valida horarios y traslapes. Por ejemplo, rechaza disponibilidades con `start_time >= end_time`, `slot_minutes` fuera de `{15,20,30,45,60}` y también rechaza traslapes con `409`. Eso puede hacer que “parezca que no guarda”, cuando en realidad sí responde con error. 

2. **Puede estar fallando porque el usuario autenticado no tiene perfil profesional real en tabla `professionals`.**
   Toda la agenda depende de `_get_professional_or_404(...)`; si la cuenta tiene rol `professional` pero no registro asociado en `professionals`, todas esas operaciones van a fallar. Eso se ve porque cada endpoint de agenda llama primero a `_get_professional_or_404`.  

3. **Puede haber un problema de sesión/token en runtime.**
   `useApi()` envía el Bearer token desde la cookie `access_token`, y la navegación protegida depende del bootstrap del store. Si el token no está entrando bien o la sesión está inconsistente, el CRUD va a fallar con 401/redirect.  

Cómo confirmarlo rápido:

* Abrir la pantalla de agenda.
* Abrir DevTools → Network.
* Probar una acción de crear o editar.
* Revisar el status del request:

  * `401/403`: problema de auth/rol/token.
  * `404`: problema de perfil profesional inexistente o recurso no encontrado.
  * `400/409`: problema de validación o traslape.
  * `200/201`: el backend sí guardó y el problema estaría en refresco/render del frontend.

Las pruebas mínimas que deberían responder desde backend son estas:

```bash
GET    /api/v1/professionals/me/availabilities
POST   /api/v1/professionals/me/availabilities
PUT    /api/v1/professionals/me/availabilities/{id}
DELETE /api/v1/professionals/me/availabilities/{id}

GET    /api/v1/professionals/me/time-blocks
POST   /api/v1/professionals/me/time-blocks
PUT    /api/v1/professionals/me/time-blocks/{id}
DELETE /api/v1/professionals/me/time-blocks/{id}
```

Un payload válido de disponibilidad sería:

```json
{
  "weekday": 0,
  "modality_code": "in_person_consultorio",
  "start_time": "08:00",
  "end_time": "12:00",
  "slot_minutes": 30
}
```

Y un payload válido de bloqueo puntual:

```json
{
  "starts_at": "2026-04-09T10:00:00",
  "ends_at": "2026-04-09T11:00:00",
  "reason": "Reunión",
  "block_type": "manual_block"
}
```
