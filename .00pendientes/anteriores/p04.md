Sí. Para darle **realismo útil** y dejar **cerrado este tema**, conviene cerrar el bloque que hoy todavía impide que la reserva se sienta real: **la configuración operativa del profesional**.

El proyecto ya tiene bastante adelantado el lado paciente: ya existen endpoints públicos para buscar profesionales, ver detalle y consultar slots, además del endpoint de reserva para paciente . También ya existen las pantallas de paciente para listado de profesionales, detalle y reserva  . Incluso el cálculo de slots ya está implementado en `SlotService`, excluyendo horarios bloqueados y citas ocupadas .

Lo que falta para que todo eso se vuelva realmente útil es que el profesional pueda, de forma completa, **configurar su perfil público, sus horarios semanales y sus bloqueos puntuales**. Hoy el backend del profesional ya tiene piezas básicas para perfil, disponibilidades y bloqueos, pero todavía está demasiado simple y sin el endurecimiento suficiente; además el layout profesional no expone una pantalla de agenda/perfil que cierre ese flujo operativo   .
Por eso la propuesta es esta:

**Tema a cerrar definitivamente:**
**Módulo “Perfil y Agenda del Profesional”**
Con esto, el profesional deja configurado:

* nombre público,
* perfil visible,
* precio referencial público,
* disponibilidad semanal,
* bloqueos puntuales.

Y desde ese momento el flujo paciente ya puede vivir sobre una base real sin volver a tocar este bloque.

Además, no hace falta tocar `main.py`, porque el router `professional_self_router` ya está registrado en la app .

# 1. Impacto y Riesgos

Este cambio toca solo el eje **profesional → configuración operativa**, sin rehacer el flujo paciente que ya existe. También evita tocar pagos, moderación y clínica, que ya tienen sus propios routers y pantallas separadas .

Riesgos principales:

1. Si se modifica mal `professional_self.py`, se puede romper la reserva paciente porque los slots dependen de disponibilidades y bloqueos.
2. Si no se validan traslapes, el profesional podría generar agendas inconsistentes.
3. Si no se audita, después será difícil rastrear quién cambió perfil, horarios o bloqueos.
4. Si se mezcla este cambio con precios operativos/pagos, el alcance se vuelve difuso y este tema vuelve a abrirse.

Protección aplicada:

* Se deja el cambio concentrado en un solo módulo.
* Se endurecen validaciones de disponibilidades.
* Se agrega auditoría con el modelo real `AuditEvent`, usando `actor_user_id`, `actor_role_code`, `resource_type` y `resource_id`, que son los campos correctos del modelo actual .
* Se evita tocar el flujo paciente ya existente.
* Se evita tocar el flujo de pagos, que ya tiene su propia pantalla de precios del profesional en otro módulo .

def _normalize_hms(value: str) -> str:
    raw = (value or "").strip()
    if not raw:
        raise HTTPException(status_code=400, detail="Time value is required")

    parts = raw.split(":")
    if len(parts) == 2:
        hh, mm = parts
        ss = "00"
    elif len(parts) == 3:
        hh, mm, ss = parts
    else:
        raise HTTPException(status_code=400, detail="Time must use HH:MM or HH:MM:SS format")

    try:
        hh_i = int(hh)
        mm_i = int(mm)
        ss_i = int(ss)
        parsed = time(hh_i, mm_i, ss_i)
    except ValueError as exc:
        raise HTTPException(status_code=400, detail=f"Invalid time value: {raw}") from exc

    return parsed.strftime("%H:%M:%S")


def _parse_hms(value: str) -> time:
    return time.fromisoformat(_normalize_hms(value))


def _ranges_overlap(start_a: time, end_a: time, start_b: time, end_b: time) -> bool:
    return start_a < end_b and start_b < end_a

async def _validate_availability_payload(
    db: AsyncSession,
    professional_id: str,
    payload: AvailabilityPayload,
    exclude_id: Optional[str] = None,
) -> Dict[str, Any]:
    start_time_normalized = _normalize_hms(payload.start_time)
    end_time_normalized = _normalize_hms(payload.end_time)

    start_parsed = _parse_hms(start_time_normalized)
    end_parsed = _parse_hms(end_time_normalized)

    if start_parsed >= end_parsed:
        raise HTTPException(status_code=400, detail="start_time must be earlier than end_time")

    if payload.slot_minutes not in VALID_SLOT_MINUTES:
        raise HTTPException(
            status_code=400,
            detail="slot_minutes must be one of: 15, 20, 30, 45, 60",
        )

    await _ensure_modality_allowed(db, professional_id, payload.modality_code)

    existing_result = await db.execute(
        select(ProfessionalAvailability).where(
            ProfessionalAvailability.professional_id == professional_id,
            ProfessionalAvailability.weekday == payload.weekday,
            ProfessionalAvailability.modality_code == payload.modality_code,
            ProfessionalAvailability.deleted_at.is_(None),
        )
    )
    existing_rows = existing_result.scalars().all()

    for row in existing_rows:
        if exclude_id and str(row.id) == str(exclude_id):
            continue

        row_start = _parse_hms(row.start_time)
        row_end = _parse_hms(row.end_time)

        if _ranges_overlap(start_parsed, end_parsed, row_start, row_end):
            raise HTTPException(
                status_code=409,
                detail="Availability overlaps with another active availability for the same weekday and modality",
            )

    return {
        "start_time": start_time_normalized,
        "end_time": end_time_normalized,
    }

# 4. Pruebas de Verificación y Regresión

Conviene probar esto como checklist técnico:

## Verificación de lo nuevo

1. Iniciar sesión con un usuario profesional.
2. Verificar que en el menú lateral aparezca **Agenda**.
3. Entrar a `/professional/agenda`.
4. Guardar:

   * nombre público,
   * título público,
   * biografía,
   * ciudad/provincia,
   * activar `is_public`.
5. Crear una disponibilidad semanal.
6. Intentar crear otra disponibilidad que se traslape en el mismo día/modalidad y verificar que falle con `409`.
7. Crear un bloqueo puntual.
8. Editar una disponibilidad.
9. Editar un bloqueo.
10. Eliminar ambos.

## Regresión obligatoria

Estas funciones antiguas deben seguir funcionando igual:

1. **Búsqueda pública de profesionales**
   Ya existe y no debe romperse .
   Debe abrirse `/patient/professionals` y listar profesionales.

2. **Detalle y reserva del profesional**
   Ya existe en frontend paciente  .
   Debe seguir mostrando slots y permitir reserva.

3. **Cálculo de slots**
   Sigue dependiendo de `SlotService`, que ya excluye bloqueos y citas ocupadas .
   Debe verificarse que un bloqueo nuevo quite slots visibles.

4. **Pantalla de precios del profesional**
   Ya existe y no debe alterarse .

5. **Listado de citas del profesional**
   Debe seguir respondiendo desde `/api/v1/professionals/me/appointments`.

# 5. Plan de Reversión

Si algo sale mal, la reversión debe ser inmediata.

## Restaurar archivos

```bash
cp backend_api/app/routers/professional_self.py.bak backend_api/app/routers/professional_self.py
cp web_frontend/layouts/professional.vue.bak web_frontend/layouts/professional.vue
cp web_frontend/pages/professional/dashboard.vue.bak web_frontend/pages/professional/dashboard.vue

rm -f web_frontend/composables/useProfessionalAgenda.ts
rm -f web_frontend/pages/professional/agenda.vue
```

## Reconstruir

```bash
docker compose -f infra/docker-compose.yml down
docker compose -f infra/docker-compose.yml up -d --build
```

## Reversión con Git

```bash
git reset --hard HEAD~1
docker compose -f infra/docker-compose.yml down
docker compose -f infra/docker-compose.yml up -d --build
```

Con este bloque, el tema de “realismo útil” queda aterrizado donde más valor genera: **el profesional por fin configura lo que hace posible la reserva real**, mientras el flujo paciente que ya existe puede aprovecharlo sin reabrir este mismo tema  .
