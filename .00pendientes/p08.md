ACTUAR COMO DESARROLLADOR BACKEND SENIOR Y MODIFICAR SOLO EL FLUJO DEL PROFESIONAL EN EL ARCHIVO:

- backend_api/app/routers/professional_self.py

OBJETIVO
Dejar funcional el flujo completo del profesional para:
1. consultar su perfil,
2. actualizar su perfil mínimo,
3. subir documentos,
4. listar documentos,
5. eliminar documentos,
6. enviar su solicitud a validación.

CONTEXTO ACTUAL
Ya existen endpoints en este router para:
- PATCH /me
- POST /me/documents
- GET /me/documents
- DELETE /me/documents/{doc_id}
- POST /me/submit-verification

Pero hoy la carga de documentos está incompleta:
- solo calcula hash en memoria,
- genera un file_id artificial,
- crea el ProfessionalDocument,
- pero no guarda el archivo físicamente,
- no crea registro real en la tabla files,
- no fuerza todavía el flujo fuerte de validación documental.

ALCANCE EXACTO
Modificar únicamente professional_self.py para que haga esto:

A. GET /me
- Debe devolver el profesional autenticado.
- Debe seguir funcionando con JWT actual.

B. PATCH /me
- Debe permitir actualizar solo estos campos:
  - public_display_name
  - professional_type
  - bio_public
  - years_experience
  - languages_json
  - public_slug
- No debe permitir cambiar:
  - id
  - user_id
  - person_id
- Debe dejar auditoría.

C. POST /me/documents
- Debe recibir:
  - document_type
  - file
- Debe resolver al profesional autenticado.
- Debe delegar el guardado físico a un servicio nuevo de archivos.
- Debe crear también registro real en la tabla files.
- Debe crear o reemplazar lógicamente el ProfessionalDocument.
- Si ya existe un documento del mismo tipo para ese profesional:
  - no se debe borrar físico directo,
  - debe marcarse deleted_at / deleted_by,
  - el archivo viejo debe moverse a carpeta lógica de eliminados.
- Para document_type = degree:
  - solo se debe permitir PDF.
- review_status inicial:
  - pending
- Debe devolver:
  - id del documento
  - document_type
  - file_id
  - original_filename
  - mime_type
  - sha256
  - review_status
  - uploaded_at
  - download_url

D. GET /me/documents
- Debe devolver solo documentos no eliminados del profesional autenticado.
- Debe incluir:
  - id
  - professional_id
  - document_type
  - file_id
  - original_filename
  - mime_type
  - sha256
  - review_status
  - review_notes
  - uploaded_at
  - download_url

E. DELETE /me/documents/{doc_id}
- Debe hacer soft delete.
- Debe marcar también eliminado el registro files asociado.
- Debe mover el archivo físico a _deleted.
- Debe registrar auditoría.

F. POST /me/submit-verification
- Debe validar que exista al menos un documento activo de tipo degree.
- Debe validar que el perfil mínimo esté completo:
  - public_display_name
  - professional_type
- Si falta algo, debe devolver 400.
- Si ya existe solicitud en estado submitted o under_review, debe devolver 409.
- Debe crear VerificationRequest.
- Debe crear VerificationEvent tipo SUBMITTED.
- Debe cambiar:
  - professional.onboarding_status = SUBMITTED
  - professional.status = PENDING_REVIEW
  - professional.is_public_profile_enabled = False
- Debe registrar auditoría.

REGLAS DE NEGOCIO OBLIGATORIAS
- Un profesional no puede quedar público ni activo sin validación documental manual aprobada.
- El documento degree se trata como el PDF principal del título.
- No se debe borrar físicamente nada crítico.
- Debe existir trazabilidad de actor y cambio.

NO HACER
- No tocar otros routers.
- No tocar auth.
- No tocar clinical.
- No integrar aún SENESCYT automática.
- No hacer scraping.
- No crear lógica nueva en frontend todavía.
