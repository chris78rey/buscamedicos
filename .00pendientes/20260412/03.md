No parece que el problema principal sea que `config.public.apiBase` llegue vacío.

Lo que sí se ve en el snapshot actual es esto:

* `nuxt.config.ts` ya define `runtimeConfig.public.apiBase` con fallback a `http://localhost:8000/api/v1`, así que en condiciones normales **no debería quedar vacío**. 
* En `web_frontend/pages/admin/verification/[id].vue` el botón del PDF usa directamente `:href="doc.download_url"`. No se ve una normalización previa en ese punto.
* El backend arma `download_url` usando `_download_url(str(doc.file_id))`, y por el síntoma reportado ese valor está saliendo relativo, tipo `/files/[ID]/download`, no absoluto con `http://localhost:8000/api/v1/...`. 

Entonces el diagnóstico más conservador es este:

1. el frontend **sí** está consumiendo `doc.download_url` tal como viene del backend;
2. el backend probablemente está devolviendo una ruta relativa; 
3. al poner esa ruta relativa en un `href`, el navegador la resuelve contra `localhost:3000`, no contra `localhost:8000`;
4. por eso el 404 aparece en el frontend aunque la URL real del backend funcione.

En otras palabras: la desconexión más probable no es Nuxt “vacío”, sino que el `href` está recibiendo una ruta relativa y el navegador la interpreta contra el host de la web.

  return value.endsWith('/') ? value.slice(0, -1) : value
}

function resolveDocumentUrl(downloadUrl?: string | null): string {
  const raw = (downloadUrl ?? '').trim()
  if (!raw) {
    return '#'
  }

  if (raw.startsWith('http://') || raw.startsWith('https://')) {
    return raw
  }

  const apiBase = normalizeApiBase(config.public.apiBase || 'http://localhost:8000/api/v1')
  const normalizedPath = raw.startsWith('/') ? raw : `/${raw}`

  if (normalizedPath.startsWith('/files/')) {

  return `${apiBase}${normalizedPath}`
